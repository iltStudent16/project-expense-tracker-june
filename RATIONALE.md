# Expense Tracker Rationale

This project uses strict TypeScript to reduce runtime risk and make refactoring safer. The strongest example is the discriminated union for `Transaction` (`income | expense`), which enforces correct branching when calculating totals and rendering amounts. Strict mode also flagged unused code and type mismatches during development, helping resolve problems earlier instead of discovering them at runtime.

I organized the code by responsibility so each layer stays focused. `src/types` contains shared contracts (`Transaction`, `Result<T,E>`), `src/data` contains selectors and persistence logic, `src/dom` handles templates and rendering, and `src/utils` contains reusable helpers such as formatting and the generic undo stack. This structure keeps business logic separate from UI wiring, which makes features easier to reason about and maintain.

Vite’s dev server improved workflow through fast startup and near-instant updates. While iterating on filters, sorting, chart rendering, and keyboard shortcuts, I could validate changes quickly without long rebuild cycles. For release confidence, `npm run build` (`tsc && vite build`) provided a strict compile + bundle check before deployment.

The most useful JavaScript/TypeScript patterns were event delegation, closures, and composable selectors. Event delegation keeps transaction list actions efficient by handling delete clicks from one listener. The closure-based `UndoStack<T>` captures state history cleanly and supports undo with `Ctrl+Z`. Selector composition (`applyFilters` then `applySort`) keeps data flow predictable. I also used `Object.entries` for chart rows, `Object.keys`/`Object.values` for category metadata, and a default + rest parameter helper (`sumWithAdjustments`) to demonstrate clear modern function design.
